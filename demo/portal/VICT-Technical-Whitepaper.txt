VICT Technical Whitepaper
Strategic Hardening of Functions-as-a-Service (FaaS)
for Security, Compliance, and Digital Sovereignty in India (2025-2026)

Version: Organized Edition
Date: March 10, 2026

====================================================================
1. Executive Summary
====================================================================

Functions-as-a-Service (FaaS) has shifted cloud security from perimeter control
to identity, configuration, runtime behavior, and data governance. In India,
this shift is amplified by legal obligations under the Digital Personal Data
Protection (DPDP) Act and RBI localization directives.

This whitepaper provides a practical framework for building a sovereign,
high-performance, and zero-trust serverless architecture. It combines:
- OWASP 2025 threat priorities
- Provider-specific architecture and performance trade-offs
- India-first compliance controls
- Advanced runtime hardening with eBPF and WebAssembly
- Forensic observability for audit-readiness

Core thesis:
Serverless hardening is no longer only a security function. In India, it is a
strategic capability that determines regulatory survivability, customer trust,
and market competitiveness.

====================================================================
2. The 2025 Serverless Threat Landscape
====================================================================

Serverless adoption continues to rise due to elasticity and pay-per-use
economics. At the same time, short-lived and high-density execution models
create attack surfaces that traditional controls often miss.

Security implications in FaaS:
- Identity misuse scales faster than network-based threats.
- A single over-privileged function can cause account-wide lateral movement.
- Ephemeral runtime behavior complicates containment and forensics.

OWASP Top 10 (2025) risk mapping for FaaS:

+------------------------------------------+--------------------------------------------+--------------------------------------------------------------+
| OWASP 2025 Category                      | Primary FaaS Risk                          | Recommended Hardening Action                                 |
+------------------------------------------+--------------------------------------------+--------------------------------------------------------------+
| A01: Broken Access Control               | Over-permissive IAM roles                  | Use resource-level ARNs and ABAC                             |
| A02: Security Misconfiguration           | Public invoke, plaintext env vars          | Enforce IaC scanning + policy-as-code                        |
| A03: Software Supply Chain Failures      | Compromised dependencies in build pipeline | Implement SCA, SBOM, signed artifacts                        |
| A05: Injection                           | Untrusted event payload injection          | Validate schemas at gateway and function boundaries          |
| A06: Insecure Design                     | Non-fail-closed event workflow logic       | Perform threat modeling on event-driven paths                |
| A07: Authentication Failures             | Token/session handling weakness            | Use managed identity and MFA-backed auth controls            |
| A09: Logging and Alerting Failures       | Missing evidence due to ephemerality       | Centralize immutable logs + anomaly detection                |
| A10: Exceptional Condition Mishandling   | Error-path leakage or control bypass       | Audit exception paths and standardize safe external errors   |
+------------------------------------------+--------------------------------------------+--------------------------------------------------------------+

Key 2025 escalation pattern:
SSRF in a function can be chained with broad IAM privileges to retrieve cloud
credentials and pivot into sensitive data stores. Least privilege and private
connectivity controls are mandatory, not optional.

====================================================================
3. Supply Chain Security Starts at the Developer Endpoint
====================================================================

Modern serverless risk begins before CI/CD:
- Malicious IDE extensions and compromised local tooling can exfiltrate secrets.
- AI-assisted coding can suggest vulnerable or stale package versions.

Required controls:
- Pin and verify dependency provenance, not only versions.
- Generate and retain SBOMs for every production artifact.
- Enforce signature verification before deploy.
- Add policy gates that fail builds on unresolved supply-chain risk.

Outcome:
You convert security from periodic review into continuous assurance.

4. Hyperscaler Comparison: Performance and Security
====================================================================

Cold starts, isolation model, concurrency behavior, and regional presence all
directly affect security architecture and compliance execution.

+--------------------------------+------------------------+-----------------------+--------------------------------------+
| Metric                         | AWS Lambda             | Azure Functions       | Google Cloud Functions (2nd Gen)     |
+--------------------------------+------------------------+-----------------------+--------------------------------------+
| Market Share (Q3 2025)         | ~29%                   | ~20%                  | ~13%                                 |
| Cold Start (Python/Node.js)    | 39ms - 1200ms          | 2000ms - 3000ms       | < 1000ms                             |
| Cold Start (Java/.NET)         | 3000ms - 14000ms       | Up to 10000ms         | < 1000ms (with CPU boost)            |
| Optimization Feature            | SnapStart              | Premium/Dedicated     | Startup CPU Boost                    |
| Concurrency Model              | 1 request/instance     | 1 request/instance    | 80-1000 requests/instance            |
| Isolation Mechanism            | Firecracker MicroVM    | Container/Hyper-V     | gVisor sandbox                       |
| India Region Availability      | Mumbai, Delhi          | Multiple local regions| Mumbai, Delhi                        |
+--------------------------------+------------------------+-----------------------+--------------------------------------+

Security-performance trade-off guidance:
- Stronger checks (encryption, token validation, DPI) may increase latency.
- Warm strategies can reduce cold starts but must avoid stale code exposure.
- Adaptive scaling and bounded concurrency can improve latency without
  weakening controls.

5. India Regulatory Requirements (DPDP + RBI)
====================================================================

5.1 DPDP Act Implications
- Implement explicit, purpose-bound consent checks before data use.
- Maintain breach response capability with 72-hour notification workflows.
- Support right-to-erasure as actual deletion, not archival-only handling.
- Keep personal data processing aligned with legal transfer constraints.

5.2 RBI Payment Data Localization (Stricter Sectoral Rule)

+-------------------------+---------------------------------------------------------------+
| RBI Checkpoint          | FaaS Technical Requirement                                    |
+-------------------------+---------------------------------------------------------------+
| In-country storage      | Geofence all payment data stores to India regions             |
| Audit rights            | Ensure contractual and operational RBI audit visibility       |
| 24-hour deletion rule   | Enforce TTL/workflow deletion for foreign-leg transient data  |
| CERT-In audit evidence  | Retain tamper-evident compliance logs and annual reports      |
+-------------------------+---------------------------------------------------------------+

Strategic implication:
A compliant Indian serverless platform must be treated as a sovereign data
system, not only a cloud deployment pattern.

====================================================================
6. Technical Hardening Blueprint
====================================================================

6.1 Identity and Least Privilege
- Assign function-specific IAM roles.
- Scope permissions to exact resources (no wildcard expansion by default).
- Apply condition keys (network boundary, MFA, context constraints).
- Automate policy synthesis from code-level SDK usage where feasible.

6.2 Secret Management
- Remove plaintext secrets from environment variables.
- Store secrets in managed vaults with encryption, rotation, and access logs.
- Use short-lived in-memory caching with strict TTL to balance latency and
  security.

6.3 Runtime Protection
- Use eBPF-driven telemetry and enforcement for syscall, file, and network
  behavior.
- Block shell spawn and unauthorized outbound connectivity.
- Enforce zero-trust behavior policies in runtime, not only at deploy time.

7. Next-Gen Runtime Strategy: Wasm + Containers
====================================================================

WebAssembly (Wasm) improves isolation granularity and startup performance for
untrusted or multi-tenant code execution.

+------------------------+---------------------------+----------------------------------+
| Feature                | Containers                | WebAssembly                      |
+------------------------+---------------------------+----------------------------------+
| Startup                | Seconds                   | Sub-millisecond possible         |
| Isolation Boundary     | OS/kernel namespace model | Bytecode sandbox + capability    |
| Default Authority      | Broad by default          | Deny-by-default capability model |
| Typical Artifact Size  | Tens/hundreds of MB       | KB to low MB range               |
+------------------------+---------------------------+----------------------------------+

Practical recommendation:
Use a hybrid model:
- Containers for broad ecosystem compatibility.
- Wasm for untrusted user logic, plugin execution, and edge-density workloads.

====================================================================
8. Forensics and Observability for Compliance
====================================================================

Serverless compliance is impossible without durable observability.

Required capabilities:
- Real-time log export from ephemeral runtime.
- Trace correlation across event chains and service boundaries.
- Immutable retention and integrity controls for audit records.
- Fast incident reconstruction for 72-hour breach reporting windows.

Reference open-source stack components:
- Collection: Fluent Bit, OpenTelemetry Collector
- Tracing/metrics/logs: SigNoz, Uptrace
- Cost-aware logging: Loki, OpenObserve
- Compliance and retention workflows: ELK, Graylog

9. India-Specific Innovation Opportunities
====================================================================

9.1 Low IT Literacy and Trust Gap
- Build security experiences that do not require high user security fluency.
- Use passkey/biometric-enabled identity bridges for sensitive services.

9.2 Sovereign Cloud Demand
- Create deployable patterns for sovereign-grade serverless on domestic
  infrastructure providers.

9.3 Sustainability Imperative
- Increase execution density and reduce idle compute waste.
- Use Wasm where suitable to lower energy and cost footprint.

11. Actionable 90-Day Roadmap
====================================================================

Phase 1 (Days 1-30): Foundation
- Inventory all functions, roles, triggers, and data stores.
- Remove wildcard IAM and public invoke paths.
- Migrate secrets to vault-based retrieval.
- Enable baseline centralized telemetry.

Phase 2 (Days 31-60): Compliance + Runtime
- Implement consent checks for personal-data workflows.
- Enforce India data residency and foreign-leg deletion controls.
- Add eBPF-driven runtime monitoring and outbound network policies.
- Add CI/CD policy gates for misconfiguration and dependency risk.

Phase 3 (Days 61-90): Innovation + Assurance
- Pilot Wasm for untrusted or high-density workloads.
- Validate breach and audit playbooks through simulation.
- Benchmark latency/cost/security trade-offs under production-like load.
- Publish control evidence pack for challenge and regulatory review.

====================================================================
12. Conclusion
====================================================================

A secure serverless architecture in India must satisfy three conditions at once:
technical resilience, legal compliance, and socio-economic fit.

Teams that integrate identity-first controls, sovereign data architecture,
supply-chain integrity, and forensic observability will outperform those that
treat security as post-deployment governance.

The strategic endpoint is clear:
Build a sovereign, high-performance, and zero-trust serverless platform that is
both innovation-ready and audit-ready.
